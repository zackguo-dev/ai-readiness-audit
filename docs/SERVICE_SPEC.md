**提供者:** Zack(郭澤華)  
**対象読者:** 提携先(Web制作会社・士業)、技術的な背景を持つ顧客担当者  
**目的:** 診断ツールの仕組みと信頼性の根拠を説明する

---

## サービス概要

Webサイトが「生成AI(ChatGPT・Gemini・Perplexityなど)にどれだけ読み取られやすいか」を、6つの技術的観点から自社開発ツールで診断し、優先度付きの改善レポートを納品するサービス。

**他社との違い:**
- 診断ツールを自社開発しているため、診断根拠を全項目で説明できる
- 効果が未確認の施策(llms.txtなど)は限界も含めて正直に説明する
- 診断から改善実装まで同一の技術者が一貫対応(伝言ゲームなし)

---

## 診断の仕組み

### データ取得

診断対象URLに対して以下を順次取得(各リクエスト間に1秒以上の待機):

| 取得対象 | 目的 |
|---|---|
| HTML本文(静的) | JS依存度・セマンティック構造・構造化データの解析 |
| robots.txt | AIボットの許可・禁止設定の確認 |
| llms.txt / llms-full.txt | AI向けガイドファイルの有無 |

Playwright(ヘッドレスブラウザ)によるJS実行後のHTMLも取得し、静的HTMLと比較することでJS依存度を実測する。

### 診断項目と根拠

**① AIボットアクセス可否(配点: 最大25点)**

対象の8ボット(GPTBot / ClaudeBot / Claude-Web / PerplexityBot / Google-Extended / CCBot / Bytespider / meta-externalagent)に対して:
1. robots.txtの許可・禁止設定を解析
2. 各ボットのUser-Agentを用いたHEADリクエストを実際に送信し、HTTP 403/429などのブロックを検出

※ この方式により「robots.txtでは許可しているが、WAF/CDNが実際には弾いている」という運営者が気づきにくい問題を検出できる。

**② llms.txt整備(配点: 最大10点)**

llms.txtおよびllms-full.txtの存在・フォーマットを確認。  
**重要:** 現時点で主要AIがこのファイルを参照することは限定的。本項目のウェイトを低く設定しているのはこのためで、「設置すればAIに読まれる」という誤解を避けている。

**③ JavaScript依存度(配点: 最大30点)**

```
JS依存率 = (JS実行後のテキスト量 - 静的HTML取得時のテキスト量) / JS実行後のテキスト量
```

多くのAIクローラーはJavaScriptを実行しないか、実行が不完全なため、JS依存度が高いサイトはAIから「本文がほぼ空のページ」に見える。本項目を最重要(配点最大)としている理由がここにある。

**④ 構造化データ(配点: 最大20点)**

JSON-LD形式のSchema.orgマークアップを抽出し、実装タイプ(Organization / Article / Product / FAQPageなど)と、各タイプの必須プロパティ欠落を検出。

**⑤ セマンティックHTML(配点: 最大10点)**

h1の有無・見出し階層のスキップ・画像altの充足率・meta description・OGPタグを確認。AIはHTMLの構造を読んでページの主題と信頼性を判断する。

**⑥ 鮮度シグナル(配点: 最大5点)**

公開日・更新日のマークアップ(OGP article:published_time / `<time datetime>` / JSON-LD datePublished)とsitemap.xmlの有無を確認。AIは日付情報を参照して情報の新鮮さを判断する。

## 納品物とレポートの構成

診断1回ごとに以下の3形式を自動生成し、`results/{ドメイン}/{タイムスタンプ}/` に保存・納品する。

### 1. PDFレポート(A4・3ページ) — 顧客納品・印刷用

| ページ | 内容 |
|---|---|
| 1. サマリー | ブランドヘッダー、総合スコア+ステータスバッジ、3カード(重大な問題/要改善/良好の項目数)、6項目のスコア表(バー+点数) |
| 2. 診断詳細 — 項目別所見 | 各診断項目のステータスバッジと所見 |
| 3. 改善アクション — 優先度別ロードマップ | 高/中/低の3段階に分類した改善提案+「本診断について」の注記 |

### 2. インタラクティブHTMLレポート — ブラウザ閲覧用

サマリー/詳細の2タブ構成。円形スコアインジケーターを表示し、各診断項目をクリックすると所見と改善提案が開閉する。

### 3. JSON — 機械可読の生データ

再診断時のビフォーアフター比較・履歴管理に使用する。

### スコアの表示基準

総合スコアのステータス表示:

| 総合スコア | 表示 |
|---|---|
| 70〜100 | 良好 |
| 50〜69 | 要改善 |
| 40〜49 | 問題あり |
| 0〜39 | 重大 |

項目別スコアの色分け(3段階): 0〜39 = 赤(重大) / 40〜69 = 琥珀(要改善) / 70〜100 = 緑(良好)

### 改善アクションの優先度分類

| 優先度 | 分類基準 |
|---|---|
| 高 — 早急に対応 | スコア40未満の項目に関する、制作会社へ依頼する改善 |
| 中 — 1〜3ヶ月以内に対応 | スコア40〜69の項目の制作会社依頼系の改善、および自社で対応できる改善すべて |
| 低 — 将来への備え | llms.txtなど、現時点で効果が未確認の将来投資 |

各改善提案には「自社で対応可」または「Webサイトの制作会社へ依頼(目安:○○円)」の区分を付記する。

**読者設定:** 技術者でない中小企業のマーケ担当者・経営者。専門用語には補足を付記。

## 診断の限界(正直に記載)

以下は本ツールが判定しないこと:

- コンテンツの品質・正確性の評価
- 検索順位への影響の予測
- AIが実際に当該サイトを引用するかどうかの保証
- Playwrightが動作しない環境でのJS依存度測定(その場合「動的検証不可」と明記)
